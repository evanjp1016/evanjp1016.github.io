2017 - 2019

Prototyping and developing a cybersecurity dashboard with Merit Network

AMON (All-packet Monitor) is a project of Merit Network’sA non-profit organization that provides computer networking services to educational, government, health care, and non-profit organizations primarily in Michigan Research & Development team. AMON is low-cost, high-quality network monitoring software that allows network operators to observe real-time internet traffic from their network and mitigate potential cyber attacks as they occur.

Overview

Context

Full-time internship, and later part-time position

Duties

Prototyping, front-end development

Team

6-8 software engineers and researchers

Problem

To monitor the status and activity of devices connected to a computer networkA collection of connected computers and devices which can share resource like an internet connection, printer, or application all linked together wirelessly or via communication devices and transmission media [source], a network operatorThe entity that has administrative rights to manage a network uses network monitoring tools.

With many standard network monitoring tools, malicious network activity, such as a cyber attack, is often not noticed until after the damage has been done. This is due to a multi-minute delay introduced while these tools sample and compress internet packets to analyze traffic before network engineers can see any network activity dataRecorded data regarding activity to and from computers and devices within a network, including internet traffic data between source and destination IP addresses.

Goal

Prior to my arrival, the team developed back-end network monitoring software, known as AMON, that collects, analyzes, and sorts all incoming internet packets from a network in real-time to eliminate the multi-minute delay before viewing network activity data.

Upon my arrival, our team’s goal was to develop and start testing a client-facing dashboard for AMON. This dashboard was meant to:

  1. visualize real-time network activity data
  2. notify network operators of potential cyber attacks as they happen
  3. provide informative data recorded during these potential attacks to help mitigate any damage

My role

My role was to develop prototypes, data visualizations, and front-end software to help create AMON’s client-facing dashboard as well as write software documentation to help network operators install and begin using AMON.

Starting with prototypes

Visualizing live network traffic

Although I was not a member of the AMON team when it was created, the cornerstone for this project was the IP Pair, a data visualization of real-time internet traffic within a network as well as an identifier for potential malicious activity.

The IP Pair functions as a heat-map which updates every three seconds visualizing internet traffic volume shared between source IP addressesInternet Protocol address is a “numerical label assigned to each device connected to a computer network that uses the Internet Protocol for communication" [source] and destination IP addresses (where at least the source or destination is within the network being monitored). The x-axis displays source binssorted and hashed collections of source or destination IP addresses and the y-axis displays destination bins, while lighter and darker tiles correspond to lower and higher traffic between these source-destination pairs.

Caption

Shown below, a dark horizontal line across the IP Pair indicates a high impact event where many source IP's target a single destination IP, likely a DDoS attackA cyber-attack in which the perpetrator seeks to make a machine or network resource unavailable by flooding the targeted machine or resource with superfluous requests from many different sources in an attempt to overload systems and prevent legitimate requests from being fulfilled [source]. When this phenomenon is seen, network operators can begin to address the issue without delay in order to prevent further damage.

Prototyping new features (where I start)

The IP Pair did have limitations. The only readable data was the source IP, destination IP, and traffic volume displayed while hovering over a tile on the IP Pair. This left room for misidentification of malicious activity. In addition, the IP Pair was basically useless if somebody wasn’t watching it at all times.

My early role on the AMON team was to prototype new features that our team worked together to ideate. The prototypes I created worked to address the IP Pair’s limitations, often taking the shape of new interactive data visualizations, data tables, or adaptations of the existing IP Pair.

Rapidly coding live prototypes

The most effective way for us to learn how new features would perform with real data was to actually use real data. Being a team within Merit Network, an Internet Service Provider and network operator, we were able to populate our prototypes with real, live internet traffic data from our own network.

This, however, eliminated the possibility of using standard design prototyping tools like InVision or Adobe XD. Instead, I chose to code functional prototypes with Python and two accompanying Python libraries, Pandas and Bokeh.

These prototypes were tested within our team to answer a simple question: does this feature help me understand what is happening within the network? In total, I built 10 feature prototypes; 4 are shown below.

AMON’s Alpha phase

Developing the Alpha phase

My earlier prototypes cleared the way for the next phase of our project: simplifying AMON’s dashboard to a few key features and developing AMON’s Alpha phase software. This was built as a lightweight, prototypical version of our software meant for testing and feedback purposes rather than a full representation of the final product.

Our back-end developers refined our data collection and analytics foundation while I designed and developed the front-end system atop their work to construct a cohesive dashboard interface.

Three key features

1. IP Pair (including history)

Addressing our goal to visualize real-time network activity, our earliest data visualization, the IP Pair, remained the cornerstone of the project. As before, it offered real-time, heat-map visualization of network activity and it clearly visualized potential DDoS attacks.

Adapted from my earlier prototypes, a new feature of the IP Pair was the ability to adjust sliders in order to go “back in time” and view internet activity from the past.

Caption

2. Alerts

Addressing our goal to identify high-impact events (potential malicious activity) as they happen, the Alerts feature offered a data table where network operators could view a list of high-impact events. This included events where internet traffic was significantly higher than usual between source and destination IP addresses. Each event was recorded with its corresponding source or destination bin number, its start and end times, its total duration in seconds, an intensity metric, and an Event ID number.

Caption

3. Forensics

Addressing our goal to provide more information about potential attacks, the Forensics feature allowed network operators to further inspect the events recorded by the Alerts feature. By either selecting particular events on the alerts page or searching for events using their Event ID, network operators could view the data collected from that event.

Using large toggle buttons, the Forensics feature is able to display information about which IP addresses were involved in the event, the volume of traffic they exchanged, the ports they used, as well as the protocols they used. This information would help determine whether or not the event was likely malicious as well as where it came from and who it targeted.

Caption

Launching the Alpha phase

The first shared version of AMON was made available to selected clients and open-source developers in the summer of 2018. After its release, we began work on AMON’s Beta phase, rebuilding the dashboard with a more conventional structure, a focus on necessary security features, and any changes based on feedback from the Alpha release.

This version of AMON is available here on Github.

AMON’s Beta phase

Focusing on structure and security

Starting from scratch and rebuilding AMON as a full Flask-based web application allowed us to follow a more conventional Model-View-Controller structure while also focusing on security and privacy necessities.

While other team members tackled the security enhancements to our new dashboard, I worked to develop official versions of our three key features.

Lean UX: cleaning the interface

The first thing we had to consider was how the dashboard interface would change from its prototypical predecessors. As this project adopted a lean approach to UX design early on, moving into the Beta phase meant cleaning and organizing the interface to simplify the user experience without spending much time on usability testing or other UX research. For me, this meant quickly creating hi-fidelity mockups of each of our key features to guide our development efforts.

Building dynamic data visualizations with d3.js

Aligning with feedback from our Alpha release, our Beta phase data visualizations had to take on a more robust form. To do this we switched from Python data visualizations to, d3.js, a javascript visualization tool.

While Bokeh, our Python-based visualization library, offered prefabricated chart-types with limited visual design control and user interactions, d3.js works by binding data to a Document Object Model (DOM). This meant that as long as we had data to visualize, we had complete control in building and styling our charts, developing how the user could interact with them, and animating how they would change upon user input or addition of new data.

This freedom allowed us to better tailor our visualizations and interactions to the needs of network engineers as well as improve their performance, visual accessibility, and viewport responsiveness.

Data retrieval and open source development with a RESTful API

To populate our features and visualizations with live network data, we decided to build a RESTful API to retrieve data from our back-end databases. This eliminated the need for our data visualizations to connect directly to back-end databases and instead allowed them to retrieve JSON data from simple web URL’s with custom parameters.

This not only made my work creating interactive features and visualizations easier, it also opened the door to open-source development of new features. With the AMON API, anybody with a valid AMON account could retrieve network data and build their own applications and data visualizations.

Leaving the project before completion

Most of AMON’s contributors were either university-funded research students, interns, or part-time workers; all of which, including myself, would most-likely not be working on the project for more than a year or two due to limited grant-funded compensation. Knowing this, we wanted to be sure that new contributors would be able to easily pick up where we left off.

I left Merit in April 2019. While we did not have AMON’s Beta phase complete, we had made it a point throughout the entire process to ensure that clear, abundant documentation and clean, understandable code were at the foundation of how this project would continue as team members changed.
